# 実装計画

更新日: 2026-07-23

## 基本原則

- 既存機能と既存データを保全する。
- 未確認の機能を削除・置換しない。
- PDF請求書読み込みは既存機能として引き継ぐ。
- 推測値は確定値へ混ぜず「要確認」で保持する。
- 公開画面と管理用データを分離する。
- 各段階で検算と実表示確認を行う。

## Phase 0 現行資産の固定と監査

- 現在の `main` を基準点として記録する。
- 主要画面、埋込データ、ワークフロー、更新スクリプトを一覧化する。
- 6月・7月データ、通帳照合、支出分類、社会保険、経営コンサルの保存位置を特定する。
- PDF請求書読み込みの実装位置と入出力を特定する。
- 7月22日データの反映経路と未反映原因を確定する。

完了条件: 既存機能一覧、ファイル対応表、データ対応表、未確認一覧が揃う。

## Phase 1 データと表示の分離

- `index.html` 内の埋込データを構造化JSONへ移す。
- 画面描画処理とデータを分離する。
- 日別売上、商品別、時間帯別、通帳、支出、給与、社会保険を独立データとして管理する。
- 既存表示と分離後表示の数値一致を自動検証する。

完了条件: 既存画面の主要数値が分離前後で一致し、旧データが欠落しない。

## Phase 2 原本取込基盤

- 券売機ジャーナル画像
- 通帳画像
- PDF・画像請求書
- 原価資料
- 給与・人件費資料

について、原本保存、読取結果、信頼度、要確認、訂正履歴を管理する。

同一ファイルのハッシュ値を使い二重取込を防止する。

完了条件: 原本→抽出→確認→確定→集計の流れが追跡できる。

## Phase 3 経営計算エンジン

- 売上、入金、未入金
- 食材原価、商品別原価、原価率
- 給与、会社負担社会保険、実質人件費
- F/L比率
- 固定費、変動費
- 営業利益、損益分岐点
- 商品別・時間帯別・曜日別採算

を同一期間・同一定義で計算する。

完了条件: 計算式、対象期間、根拠データ、要確認額が画面から確認できる。

## Phase 4 AI経営コンサル

AIの提案は必ず以下を伴う。

- 根拠となる数値
- 比較対象
- 推奨行動
- 想定効果
- 不確実性または要確認事項

根拠のない断定や、期間の異なる数字を混ぜた利益判断は禁止する。

完了条件: 提案から根拠データへ遡れ、数字が変われば提案も再計算される。

## Phase 5 公開版と管理版の分離

公開版:

- 集計済み経営指標
- 個人情報を含まない分析

管理版:

- 原本
- 読取結果
- 訂正
- 給与明細
- 請求書
- 通帳明細
- 履歴

完了条件: GitHub Pages上へ機密情報や原本が含まれないことを確認する。

## Phase 6 運用・検証

- Mac、Windows、iPhone、Androidで確認する。
- 新規データ追加時の手順を一本化する。
- 更新前後の自動バックアップを作る。
- 失敗時に旧公開版へ戻せるようにする。
- 月曜日は原則休業だが、実績データがある場合は実績を優先する。

完了条件: 新しい原本を投入し、確認・確定後にダッシュボードへ反映され、数値検算と公開確認まで完了する。

## 直近の実施順

1. GitHub履歴とワークフローからPDF請求書読込の所在を確定する。
2. 7月22日データを既存構造のまま正しく反映し、公開表示を確認する。
3. 現行 `index.html` の全データ区分を一覧化する。
4. 分離前の基準値テストを作る。
5. JSON分離を小単位で開始する。

大規模な一括置換は行わない。各段階で既存結果と一致することを確認してから次へ進む。
