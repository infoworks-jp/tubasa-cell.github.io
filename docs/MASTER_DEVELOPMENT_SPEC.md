# 味一番つばさ 経営管理・AI経営参謀システム 完成仕様

最終更新: 2026-07-23

## 1. 開発方針

このシステムは、既存のGitHub Pages、既存データ、既存のPDF請求書読込、券売機ジャーナル読込、通帳読込、ダッシュボードを破棄せずに引き継ぎ、統合・強化する。

未検証の処理を「完成」と扱わない。完成判定は、実データを投入し、数値照合、画面表示、再読込、公開環境での動作確認まで完了した場合に限る。

既存データは推測で補完しない。不明値は「要確認」として保持し、原本と抽出値と修正履歴を分離して保存する。

## 2. 初期完成時に必須の入力

以下は将来機能ではなく、初期完成条件に含む。

- 券売機ジャーナル画像
- 通帳画像
- PDF請求書
- 画像形式の請求書・領収書
- 原価資料（PDF、Excel、CSV、画像）
- 人件費資料
- 社会保険等の会社負担設定
- 固定費・変動費資料

## 3. 原本管理

すべての取込データについて以下を保持する。

- 原本ファイル
- ファイル名
- 取込日時
- 対象期間
- 文書種別
- 抽出結果
- 読取信頼度
- 要確認項目
- 手修正前の値
- 手修正後の値
- 修正者・修正日時・修正理由

同じ原本の二重取込を検知する。

## 4. 券売機ジャーナル処理

- 営業日
- 売上高
- 客数
- 客単価
- 発券数
- 精算数・精算金額
- 商品別数量・売上
- 時間帯別数量・売上
- 昼・夜・深夜区分
- 休業日
- 読取不能項目

月曜日は当分休業扱いを既定値とするが、実績データが存在する場合は実績を優先する。

売上合計、商品別合計、時間帯別合計、発券・精算の関係を相互検算し、差異を表示する。

## 5. 請求書処理（既存機能として引継ぎ）

PDF請求書読込は既存設計であり、初期完成時から利用可能でなければならない。

抽出項目:

- 発行元
- 請求先
- 請求書番号
- 請求日
- 対象期間
- 支払期限
- 税抜金額
- 消費税
- 税込金額
- 品目
- 数量
- 単価
- 軽減税率・標準税率
- 振込先
- 適格請求書発行事業者番号
- 原価・経費区分

請求書は仕入、外注、経費、固定費に分類し、通帳支払と照合する。

## 6. 通帳処理

- 取引日
- 摘要
- 入金
- 出金
- 残高
- 売上入金
- 仕入支払
- 給与支払
- 社会保険支払
- 税金
- 借入・返済
- その他

売上日は手書き記載がある場合、その日付を優先する。売上と入金の時間差を考慮し、完全一致、候補一致、未照合に分類する。

## 7. 商品・原価管理

商品マスターに以下を持つ。

- 商品コード
- 商品名
- カテゴリ
- 販売価格
- 税区分
- レシピ
- 食材別使用量
- 食材単価
- 包材原価
- 商品原価
- 原価率
- 粗利益額
- 粗利益率
- 有効開始日・終了日

価格・原価の変更履歴を保持し、販売日の有効単価で計算する。

「味噌ラーメン」は店舗表現として「究極の味噌」を使用する。

## 8. 人件費管理

実質人件費は給与額だけでなく、会社負担を含めて計算する。

- 基本給
- 時給・日給
- 残業代
- 深夜割増
- 休日割増
- 交通費
- 賞与
- 雇用保険会社負担
- 健康保険会社負担

- 子ども・子育て拠出金
- 労災保険
- 有給休暇引当相当
- その他法定福利費

料率は設定画面で年度別に変更可能とし、計算根拠を表示する。個人情報を含む給与明細は公開画面に出さない。

## 10. AI経営参謀

AIコメントは数字の言い換えではなく、根拠、優先順位、想定効果、実施難易度を伴う提案とする。

出力例:

- 最優先課題
- 悪化・改善の原因
- 商品構成の問題
- 時間帯別の人員過不足
- 原価上昇の影響
- 値上げ候補
- 販売強化候補
- 廃止・縮小候補
- ビール・餃子等のセット提案
- シフト改善案
- 損益分岐点達成に必要な売上・客数
- 今月残日数で必要な日商
- 予測幅と不確実性

各提案に「使用した数値」「比較期間」「計算式」「前提」を表示する。根拠データが足りない提案は断定せず、「要確認」とする。

## 12. 公開範囲

公開用GitHub Pagesには、個人情報、給与明細、通帳残高、口座番号、請求書原本、取引先の機密情報を掲載しない。

管理機能と公開閲覧機能を分離する。公開用データは必要項目のみを書き出す。

GitHub Pages単体では秘密情報を安全に保持できないため、管理処理を完全にブラウザだけで行う場合は、原本を外部送信せず端末内で処理する設計、または認証された管理基盤を採用する。公開リポジトリへ原本・個人情報・APIキーをコミットしない。

## 13. 自動処理の完成条件

通常運用では、利用者が指定フォルダまたは取込画面へ原本を追加すると、以下を一連で実行する。

1. ファイル種別判定
2. OCR・PDFテキスト抽出
3. データ正規化
4. 重複検査
5. 数値検算
6. マスター照合
7. 要確認抽出
8. 売上・入金・請求・支払照合
9. 原価・人件費・F/L・利益計算
10. ダッシュボード更新
11. AI経営提案更新
12. バックアップ
13. 更新履歴記録

読取不能や不一致がある場合も処理全体を停止せず、該当部分を「要確認」として残す。

## 15. 完成判定

次の全項目を満たすまで「完成」と表現しない。

- 既存機能が失われていない
- PDF請求書読込が動く
- 券売機画像読込が動く
- 通帳画像読込が動く
- 数字が原本と一致する
- 原価率、人件費率、社会保険会社負担、F/L、利益が検算済み
- AI提案が根拠付きで表示される
- 新規原本を追加するだけで通常処理が完了する
- 公開サイトが正常表示される
- 個人情報が公開されない
- 実際の運用手順書がある
- バックアップ・復元手順がある
- 利用者が実運用できることを確認した
